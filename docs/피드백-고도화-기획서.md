# 피드백 시스템 고도화 기획서

## 배경

현재 피드백 시스템은 텍스트 기반 제출만 지원하며, 수정 권한이 admin에게만 있음.
사용자 경험 개선 + 스크린샷 첨부 기능 추가 필요.

---

## 변경 범위

### 1. 피드백 수정 권한 확장

#### 현재

| 동작        | 일반 유저 | 관리자                   |
| ----------- | --------- | ------------------------ |
| 작성        | O         | O                        |
| 조회 (본인) | O         | O                        |
| 조회 (전체) | X         | O                        |
| 수정        | X         | O (status, admin_memo만) |
| 삭제        | X         | O                        |

#### 변경 후

| 동작               | 일반 유저                         | 관리자                                                |
| ------------------ | --------------------------------- | ----------------------------------------------------- |
| 작성               | O                                 | O                                                     |
| 조회 (본인)        | O                                 | O                                                     |
| 조회 (전체)        | X                                 | O                                                     |
| **수정 (본인 글)** | **O (category, message, 이미지)** | O                                                     |
| **수정 (타인 글)** | X                                 | **O (status, admin_memo, category, message, 이미지)** |
| 삭제               | X                                 | O                                                     |

#### 유저 수정 조건

- 본인 이메일(`user_email`)과 세션 이메일 일치
- 수정 가능 필드: `category`, `message`, 첨부 이미지
- status, admin_memo는 여전히 admin 전용
- 수정 시 `updated_at` 갱신

---

### 2. 이미지 첨부 기능

#### 지원 방식

1. **클립보드 붙여넣기** (Ctrl+V / Cmd+V)
   - textarea 영역에서 `paste` 이벤트 감지
   - `clipboardData.items`에서 `image/*` 타입 추출
   - 미리보기 표시 후 업로드

2. **파일 선택 업로드**
   - 버튼 클릭 → `<input type="file" accept="image/*">` 트리거
   - 복수 파일 지원 (최대 3장)

#### 이미지 저장

- **저장소**: Supabase Storage (`feedback-images` 버킷)
- **경로**: `{feedback_id}/{uuid}.{ext}`
- **허용 포맷**: PNG, JPG, JPEG, GIF, WebP
- **최대 크기**: 파일당 5MB
- **최대 개수**: 피드백당 3장

#### DB 스키마 변경

**방안: `user_feedback` 테이블에 `image_urls` 컬럼 추가**

```sql
ALTER TABLE user_feedback
ADD COLUMN image_urls text[] DEFAULT '{}';
```

- `text[]` (PostgreSQL 배열) — Supabase Storage public URL 목록
- 최대 3개 URL 저장
- 별도 테이블 불필요 (1:N 관계지만 상한이 3이므로 배열로 충분)

#### 이미지 업로드 플로우

```
사용자 Ctrl+V 또는 파일 선택
  → 클라이언트 미리보기 (ObjectURL)
  → 피드백 제출/수정 시 이미지도 함께 업로드
  → API: Supabase Storage에 업로드 → public URL 획득
  → user_feedback.image_urls에 URL 배열 저장
```

#### 이미지 삭제

- 피드백 수정 시 기존 이미지 제거 가능
- 피드백 삭제 시 Storage 파일도 함께 삭제

## API 변경

### PATCH `/api/feedback`

현재: admin만 `status`, `adminMemo` 수정 가능

변경:

```typescript
// 권한 분기
if (isAdmin(email)) {
  // admin: 모든 필드 수정 가능
  // status, adminMemo, category, message, image_urls
} else if (feedback.user_email === email) {
  // 본인: category, message, image_urls만 수정
  // status, adminMemo 변경 시도 시 403
} else {
  // 타인: 403 Forbidden
}
```

### POST `/api/feedback`

- 기존 텍스트 필드 + `image_urls` (선택) 추가

### POST `/api/feedback/upload` (신규)

- 이미지 업로드 전용 엔드포인트
- `multipart/form-data` 수신
- Supabase Storage 업로드 → public URL 반환
- 인증 필수 (로그인 유저만)

## Supabase 변경

### Migration

```sql
-- 1. image_urls 컬럼 추가
ALTER TABLE user_feedback
ADD COLUMN image_urls text[] DEFAULT '{}';

-- 2. Storage 버킷 생성 (Supabase Dashboard 또는 SQL)
INSERT INTO storage.buckets (id, name, public)
VALUES ('feedback-images', 'feedback-images', true);

-- 3. Storage RLS
CREATE POLICY "authenticated_upload"
ON storage.objects FOR INSERT
TO authenticated
WITH CHECK (bucket_id = 'feedback-images');

CREATE POLICY "public_read"
ON storage.objects FOR SELECT
USING (bucket_id = 'feedback-images');

CREATE POLICY "service_delete"
ON storage.objects FOR DELETE
TO service_role
USING (bucket_id = 'feedback-images');
```

3. **컴포넌트**: ImageUploader 신규 생성

6. **컴포넌트**: FeedbackDetailPanel 이미지 갤러리

## 리스크 & 고려사항

| 항목                | 내용                                                                 |
| ------------------- | -------------------------------------------------------------------- |
| Storage 용량        | 5MB × 3장 × N건 — 당분간 문제 없음, 모니터링 필요                    |
| 이미지 리사이즈     | 클라이언트에서 리사이즈 없이 원본 업로드 (5MB 제한으로 충분)         |
| 브라우저 호환       | Clipboard API — 모든 모던 브라우저 지원                              |
| RLS vs Service Role | Storage 업로드를 API Route에서 service_role로 처리 (RLS 복잡도 회피) |
| 수정 이력           | 수정 이력 추적 안 함 (updated_at만 갱신) — 필요시 Phase 2            |
