# 260429\_대시보드 성능 개선 및 집계 전환 계획

## Summary

- 현재 병목은 페이지 진입마다 전체 `AdRow[]`를 가져오고, 클라이언트에서 화면별 집계를 반복하는 구조다.
- 1단계는 얇고 빠른 체감 개선만 적용한다: 불필요한 재요청, 깜빡임, 전체 row 초기 fetch를 줄인다.
- 2~3단계는 로케일/작품종합/오버뷰를 Supabase 집계 기반으로 전환해 브라우저가 RAW 전체가 아닌 집계 결과만 받게 한다.
- 기존 `/api/dashboard`, Supabase schema, Explore의 `/api/query`는 즉시 삭제하지 않고 병행 유지한다.

## Key Changes

- 1단계: 빠른 체감 개선
  - `/dashboard/locale`은 서버 진입 시 전체 row가 아니라 URL의 `locale`, `start/end`, `mediums`, `goals`를 반영한 필요한 범위만 `fetchDashboardData`로 가져온다.
  - `/dashboard/titles`도 URL 필터가 있으면 서버 초기 fetch에 반영한다. 필터가 없는 기본 진입은 기존 전체 기준을 유지한다.
  - `LocaleDetailView`와 `TitleOverviewView`의 URL 동기화는 `router.replace` 대신 `window.history.replaceState`로 바꿔 App Router 재실행과 깜빡임을 줄인다.
  - 클라이언트 첫 fetch는 서버에서 받은 `initialData`와 같은 조건이면 생략한다.
  - `/api/dashboard`의 단일 슬롯 캐시는 TTL Map/LRU 캐시로 바꿔 여러 필터 조합을 동시에 유지한다.
  - `fetchFilterOptions` 결과도 서버 메모리 TTL 캐시로 감싸 페이지 이동마다 필터용 전체 row 스캔을 반복하지 않는다.
  - 이 단계에서는 전역 클라이언트 raw-data store는 만들지 않는다. 3단계에서 레거시가 될 가능성이 높기 때문이다.

- 2단계: 로케일/작품종합 집계 API 도입
  - 새 report 계층을 추가한다. 내부 구현은 기존 `QueryDefinition` + `executeQuery` + Supabase `dynamic_aggregate`를 우선 사용한다.
  - 새 API를 추가한다:
    - `GET /api/reports/locale?locale=US&startDate=...&endDate=...&mediums=...&goals=...`
    - `GET /api/reports/titles?startDate=...&endDate=...&countries=...&mediums=...&goals=...&periodMode=all|month|week&period=...`
  - 로케일 API는 KPI, 목표별 분포, 주차별 차트 2종, 작품별/소재별/매체별 테이블에 필요한 집계 payload를 한 번에 반환한다.
  - 작품종합 API는 결제 캠페인 랭킹, 가입 캠페인 랭킹, 전체 광고비 랭킹, 기간 옵션, 상단 요약 카드에 필요한 집계 payload를 반환한다.
  - UI 컴포넌트는 `AdRow[]` 직접 집계 대신 report payload를 렌더링하도록 변경한다.
  - 기존 `/api/dashboard` 기반 구현은 parity 확인 전까지 fallback/reference로 남긴다.

- 3단계: 오버뷰 집계 전환
  - 새 API를 추가한다:
    - `GET /api/reports/overview?startDate=...&endDate=...&countries=...&mediums=...&goals=...`
  - 오버뷰 KPI, 추이 차트, 매체별 성과, 로케일 카드, 요약 테이블, 랭킹은 집계 payload 기반으로 전환한다.
  - 상세 RAW 테이블과 CSV export는 별도 lazy/paginated raw endpoint로 분리한다. 오버뷰 첫 진입에서 11,000+ rows를 받지 않는다.
  - 모든 집계 페이지 전환 후 `/api/dashboard`는 export/legacy/raw-table 전용으로 축소하거나 후속 정리 계획에서 제거한다.

## Public Interfaces

- 유지:
  - 기존 `/api/dashboard`
  - 기존 `/api/query`
  - 기존 `QueryDefinition`
  - 기존 Supabase schema와 `dynamic_aggregate`
- 추가:
  - `/api/reports/locale`
  - `/api/reports/titles`
  - `/api/reports/overview`
- 공통 필터 의미:
  - `countries`, `locale`, `mediums`, `goals`는 comma-separated query string을 사용한다.
  - 날짜는 `startDate`, `endDate`를 우선 사용하고, 기존 `start`, `end` URL 표기는 UI URL에서만 유지해도 된다.
  - ROAS는 `revenue / adSpend * 100`, CTR은 `clicks / impressions * 100`, CPA는 `adSpend / signups` 기준을 유지한다.

## Test Plan

- 1단계 검증
  - `npm run typecheck`
  - 변경 파일 대상 `npx eslint ...`
  - 가능하면 `npm run build`
  - `/dashboard/locale?locale=US` 진입 시 초기 API가 전체 11,000 rows가 아닌 US 범위만 가져오는지 확인한다.
  - `TW -> US` 전환 시 데이터가 보였다가 깜빡이며 재로딩되는 현상이 줄었는지 확인한다.
  - 같은 필터 조합 반복 호출에서 `/api/dashboard`가 `X-Cache: HIT`를 안정적으로 반환하는지 확인한다.
  - `/api/filters`가 페이지 이동마다 반복적으로 전체 조회하지 않는지 응답 시간으로 확인한다.

- 2단계 검증
  - `US`, `TW`, `봄툰 KR`, `레진 KR`에서 기존 row 집계값과 새 report API 값의 KPI/차트/테이블 숫자가 일치하는지 비교한다.
  - 로케일 상세의 주차별 그래프, 목표별 표, 작품/소재/매체 탭이 기존 UI 구조를 유지하는지 확인한다.
  - 작품종합의 결제/가입/전체 랭킹이 기간 모드 변경에 따라 갱신되는지 확인한다.
  - report API 응답 크기가 기존 `/api/dashboard` 대비 크게 줄었는지 기록한다.

- 3단계 검증
  - 오버뷰 KPI, 추이, 매체별 성과, 로케일 카드 숫자가 기존 방식과 일치하는지 비교한다.
  - 오버뷰 첫 진입에서 대량 RAW row 요청이 발생하지 않는지 네트워크 탭으로 확인한다.
  - 상세 RAW 테이블/CSV export는 사용자가 해당 기능을 사용할 때만 raw 데이터를 요청하는지 확인한다.
  - Docker 빌드 및 실행 확인: `docker compose up -d --build aminsight`.

## Assumptions

- 1단계는 빠른 체감 개선용이며, 큰 구조 변경이나 UI 재설계는 하지 않는다.
- 2~3단계는 기존 Explore/query-engine 방향을 따른다. 새 집계 구조를 만들되, 두 번째 query abstraction은 만들지 않는다.
- Supabase schema 변경 없이 `dynamic_aggregate`로 가능한 범위부터 전환한다. 부족한 집계가 발견될 경우 별도 SQL/RPC 계획을 작성한다.
- 기존 `/api/dashboard`는 새 report API parity가 확인될 때까지 삭제하지 않는다.
- 캐시는 서버 메모리 TTL 기준으로 시작한다. Redis 같은 외부 캐시는 이번 계획 범위에 포함하지 않는다.
