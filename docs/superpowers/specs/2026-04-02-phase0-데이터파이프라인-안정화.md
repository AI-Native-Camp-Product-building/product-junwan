# Phase 0: 데이터 파이프라인 안정화 — 개발 기획서

**작성일**: 2026-04-02
**목표 기간**: 1주 (5영업일)
**전제 조건**: 기능정의서 v1.0 3인 전문가 리뷰 완료

---

## 1. Phase 0 목표

> **9개 시트 전체에서 실데이터가 Supabase `ad_normalized` 뷰를 통해 정상 조회되는 상태를 만든다.**

이것이 전체 대시보드 프로젝트의 선행 조건이다. 데이터 없이는 UI 작업이 불가능하고, 데이터 품질 없이는 KPI 수치가 무의미하다.

---

| 종합 시트 처리 | sheets.json에 등록됨 | 종합 시트가 다른 시트의 합산본이면 중복 문제 |

## 3. 서브에이전트 구성 제안

### 구성: 3개 에이전트 병렬 실행

│  데이터 검증            코드 보강              & 검증 UI        │
│                                                               │
│  ┌──────────┐          ┌──────────┐          ┌──────────┐    │
│  │ 9개 시트  │          │ parser   │          │ dashboard │    │
│  │ 헤더 수집 │          │ 패턴 추가 │          │ shell 수정│    │
│  │ 데이터 품질│          │ sync 안정화│         │ sync UI   │    │

### Agent 1: Data Engineer — 시트 스캔 & 데이터 검증

**목적**: 9개 시트의 실제 구조를 파악하고, 현재 파서가 커버하지 못하는 gap을 식별

**작업 내용**:
1. **시트 접근 권한 검증** — 서비스 계정으로 9개 시트에 실제 접근 시도
2. **헤더 행 수집** — 각 시트의 `header_row`에 해당하는 행을 읽어 실제 컬럼명 목록 확보
3. **데이터 샘플 수집** — 각 시트에서 데이터 행 5개 샘플링하여 타입/포맷 확인
4. **매핑 gap 리포트** — `medium_map`, `goal_map`, `creative_type_map`에 없는 raw 값 목록화
5. **날짜 포맷 검증** — `month_raw`, `date_raw`의 실제 포맷 변형 수집
6. **종합 시트 분석** — 다른 8개 시트와 데이터 중복 여부 확인

**산출물**: 시트별 스캔 리포트 (`docs/specs/시트별-데이터-스캔-결과.md`)

### Agent 2: Backend Architect — 파서/동기화 코드 보강

**목적**: Agent 1의 리포트를 기반으로 sheets-parser와 sheets-sync의 코드를 수정

**작업 내용**:
1. **`sheet_source` 시드 업데이트** — 10개 플랫폼 전체 등록 (레진JP, 벨툰 추가)
   - 기능정의서 기준: 레진KR / 봄툰KR / US / ES / DE / FR / 레진JP / 벨툰 / TH / TW
   - 현재 schema.sql: 8개만 등록 (레진JP, 벨툰 없음)
   - ⚠️ sheets.json에도 이 2개가 없음 — 시트 URL 확보 필요
2. **헤더 패턴 보강** — Agent 1이 발견한 미매핑 헤더를 `HEADER_PATTERNS`에 추가
3. **매핑 테이블 시드 보강** — 새로 발견된 `medium_raw`, `goal_raw`, `creative_type_raw` 값에 대한 INSERT 문 추가
4. **종합 시트 처리 결정** — 중복이면 `is_active: false`, 고유 데이터면 별도 로직
5. **에러 핸들링 강화** — 파싱 실패 시 상세 로그 (어떤 시트, 어떤 행, 어떤 필드)
6. **동기화 결과 요약 개선** — 시트별 매핑 성공률, 누락 필드 카운트 반환

**산출물**: 수정된 코드 파일들

### Agent 3: Frontend Developer — 대시보드 연결 & 검증 UI

**목적**: 실데이터가 UI에 정상 표시되는지 확인하고, 동기화 상태를 모니터링할 수 있는 최소 UI 추가

**작업 내용**:
1. **동기화 상태 표시** — 대시보드 헤더에 "마지막 동기화: 2026-04-02 18:00" 표시
   - `sheet_sync_latest` 뷰에서 최신 동기화 시각 조회
2. **수동 동기화 트리거 버튼** — 관리자용 Sync 버튼 (POST /api/sync 호출)
3. **시트별 동기화 상태** — 설정 페이지 또는 사이드바에 9개 시트의 상태 표시
   - 성공(초록) / 실패(빨강) / 진행중(노랑) / 미동기화(회색)
4. **데이터 없음 상태 처리** — KPI 카드/차트/테이블에 데이터가 없을 때 빈 상태 UI
5. **실데이터 바인딩 확인** — 기존 `sample.ts` 경로가 아닌 `/api/dashboard` 경로로 데이터 표시 확인

**산출물**: 수정된 컴포넌트 파일들

## 5. Phase 0 완료 기준 (Exit Criteria)

| # | 기준 | 검증 방법 |
|---|------|----------|
| 1 | 9개 시트 모두 Sheets API로 읽기 가능 | `readSheet()` 성공 |
| 2 | `POST /api/sync` 실행 시 전체 success 또는 partial (failed 0) | API 응답 확인 |
| 3 | `ad_normalized` 뷰에서 9개 시트 데이터 조회 가능 | Supabase 쿼리 |
| 4 | `medium_map` 누락 없음 (unmapped raw value 0) | ad_normalized에서 medium = raw_value인 행 확인 |
| 5 | 대시보드 UI에 실데이터 KPI/차트/테이블 표시 | 육안 확인 |
| 6 | 동기화 상태 UI에서 시트별 상태 확인 가능 | 육안 확인 |
| 7 | 종합 시트 중복 데이터 처리 완료 | 총 행수 검증 |

## 6. 주요 리스크 & 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 시트 공유 안 됨 | 해당 시트 동기화 불가 | 기획자에게 즉시 공유 요청, 나머지 시트로 우선 진행 |
| 시트별 헤더가 예상과 다름 | 파서 실패 | Agent 1이 먼저 전수 스캔하여 리스크 선제 파악 |
| 레진JP/벨툰 시트 URL 없음 | 2개 플랫폼 누락 | sheets.json에 없으므로 기획자에게 확인 필요 |
| 종합 시트가 중복 합산본 | 데이터 이중 계산 | is_active: false 처리 또는 동기화에서 제외 |
| date_raw 캐스팅 실패 | ad_normalized 뷰 에러 | CASE WHEN으로 방어 또는 text 유지 |
| 매핑 테이블에 없는 값 대량 발견 | 정규화 실패 | COALESCE fallback이 이미 있으므로, 리포트 후 점진 추가 |

## 7. 기획자 확인 필요 사항 (Blocker)

Phase 0 착수 전 반드시 확인해야 할 항목:

1. **레진JP / 벨툰 시트 URL** — `credentials/sheets.json`에 없음. 시트가 존재하는지, URL은 무엇인지 확인
2. **9개 시트 서비스 계정 공유 상태** — `pat-n8n@gen-lang-client-0537880120.iam.gserviceaccount.com`이 각 시트에 뷰어로 추가되어 있는지
3. **종합 시트의 역할** — 다른 8개 시트의 합산본인지, 아니면 고유 데이터가 있는지
4. **시트 구조 변경 가능성** — 현재 시트 구조가 확정인지, 향후 컬럼 추가/변경 예정이 있는지

---

## 8. Phase 0 → Phase 1 연결

Phase 0이 완료되면 Phase 1 (MVP 대시보드)에서 다음을 바로 시작할 수 있다:

- 사이드바 메뉴 재구성 (홈 / 플랫폼별 / 매체별)
- 날짜 범위 프리셋 필터
- 실데이터 기반 KPI 카드 강화 (목표 게이지)
- 추이 차트 탭 전환 (광고비 / 가입 / 결제 / ROAS)

Phase 0에서 확보한 데이터 품질과 파이프라인 안정성이 Phase 1의 속도를 결정한다.
